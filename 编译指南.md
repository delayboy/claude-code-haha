| **Windows 二开友好度**     | ⚠️ 中——VS C++ Build Tools 是硬门槛                    |

### 建议

**如果你的目标是二开桌面端（修改 UI / 业务逻辑），而不是改 native 层**：

1. **可以跳过 MSVC**——只跑 `cd desktop && bun install && bun run dev`，node-pty 在 dev 模式下用 prebuild 即可（前提是 Electron 版本匹配）
2. **不要碰打包流程**（`build:sidecars` + `electron-builder`），打包才真正需要 MSVC
3. **跨机迁移**时，把源码 + `~/.bun/install/cache/` 一起带，能省 80% 重新下载时间

**如果你需要出 release 包**：

1. Windows 必须装 VS 2022 Build Tools C++ workload
2. 这会和你现有的 C++ 项目共存（VS 支持多版本并存），但磁盘占用增加 5-7GB
3. **建议用 GitHub Actions 出包**（项目本来就支持，见 [.github/workflows/release-desktop.yml](.github/workflows/release-desktop.yml)），不在本地打包

**总体建议**：项目本身工程化做得不错（清晰的脚本、文档完整），二开是可行的，**主要风险点是 Windows 上的 MSVC 依赖**。如果你接受"dev 模式本地跑、release 走 CI"的工作流，移植难度是可控的。



# 根目录src代码的作用

## 分析结果

必须执行 `bun run build:sidecars`  编译服务端，代码才能起作用

`src/services/api/claude.ts` 属于**根包**（CLI 程序），也就是 "Claude Code Haha" 本体。

### 它编译到哪个程序？

这个文件**不需要单独编译**。根包（CLI）不使用传统的构建工具（没有 webpack、vite、tsc 编译步骤），而是由 **Bun 运行时直接执行 TypeScript 源码**。Bun 在运行时会即时（JIT）转译所有导入的 `.ts`/`.tsx` 模块。你执行任意入口点时，该文件会作为依赖被动态加载。

### 如何"编译"并运行？

根包没有 `build` 脚本，直接运行源码：

```bash
# 方式 1：通过 shell 脚本启动（推荐，会自动加载 .env 和 preload）
bun run ./bin/claude-haha

# 方式 2：直接运行 CLI 入口
bun run ./src/entrypoints/cli.tsx

# 方式 3：使用 npm scripts
bun run start
```

### 编译到桌面客户端（Electron）？

桌面客户端 (`desktop/`) 是独立的 Electron 应用，有自己的 Vite + TypeScript 构建链。如果要编译桌面版：

```bash
cd desktop
bun run build           # 构建 React 前端
bun run electron:build  # 构建 Electron 主进程 + 前端
bun run electron:package # 打包为 .exe
```

但注意桌面客户端可能不会直接使用 `src/services/api/claude.ts`，它有自己的 API 抽象层。

### 总结

| 程序     | 编译方式                       | 命令                                   |
| -------- | ------------------------------ | -------------------------------------- |
| CLI 本体 | Bun 运行时直接执行，无编译步骤 | `bun run start`                        |
| 桌面版   | Vite + tsc 构建                | `cd desktop && bun run electron:build` |

# 修复GLM ai端口调用超时问题

所有三处修改都已完成。以下是总结：

---

## 修改总结

### 修改了 3 个文件

| 文件                                                         | 位置                         | 作用                                                   |
| ------------------------------------------------------------ | ---------------------------- | ------------------------------------------------------ |
| `node_modules/@anthropic-ai/sdk/core/streaming.js` (line 65-70) | SDK 编译产物，实际运行时加载 | **核心修复**：检测缺少 `event: error` 行的数据错误事件 |
| `node_modules/@anthropic-ai/sdk/src/core/streaming.ts` (line 86-94) | SDK 源码                     | 同上，保证源码一致性                                   |
| `src/services/api/claude.ts` (line 2121-2134)                | 流消费层                     | 防御层：万一事件穿透了 SDK 层，在此处兜底              |

### 修复原理

GLM 返回的 SSE 错误：
```
data: {"type":"error","error":{"type":"overloaded_error",...}}
```
缺少 `event: error` 行 → SDK 的 `SSEDecoder` 把 `event` 设为 `null` → 所有 `sse.event === 'xxx'` 检查都匹配不上 → **错误被静默丢弃**。

修复后新增的检查逻辑：
```
if (!sse.event) {                    // 没有 event 行
  parsed = JSON.parse(sse.data)     // 解析 data
  if (parsed.type === 'error') {    // 发现是错误事件
    throw APIError(...)             // 正确抛出，进入重试流程
  }
}
```

### 如何让修改生效

**Sidecar 是编译好的二进制**，修改后需要重新编译：

```bash
cd desktop
bun run build:sidecars
```

然后重启桌面应用即可。`electron:dev` 模式不会自动重建 Sidecar，必须手动执行。

### 注意事项

直接修改 `node_modules` 是临时方案，执行 `bun install` 会丢失修改。建议后续用 `bun patch` 或 `patch-package` 持久化 SDK 补丁。
